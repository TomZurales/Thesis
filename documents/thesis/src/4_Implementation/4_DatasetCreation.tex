\subsection{Dataset Creation}

Datasets were created to test the performance of the observability model and to evaluate the performance and impact of libVBEE on SLAM systems. A combination of synthetic and real-world data was used, with the goal of providing a diverse set of scenarios for testing.

\subsection{Observability Scenario Dataset}
\label{sec:observability_scenario_dataset}

The observability scenario dataset is a synthetic collection of unique observability scenarios designed to evaluate the performance of the observability model. Each dataset entry, referred to as an ``observability scenario,'' is represented as a spherical volume with a single map point at the center. Scenarios contain two types of barriers: keep-out zones and obstructions. Keep-out zones represent viewpoints relative to the map point that the camera cannot access (for example, below ground). Obstructions represent objects that the camera can move behind but that block the observation of the map point (such as a wall shared with another room).

An observability scenario is defined by selecting a barrier type for each of the six directions: $x^+, y^+, z^+, x^-, y^-, z^-$. Barriers can be placed either \textit{near} (touching the map point) or \textit{far} (at a distance of $\dmax/2$ from the map point). For each direction, one of the following options is chosen:
\begin{itemize}
    \item (0) No obstruction
    \item (1) Near obstruction
    \item (2) Far obstruction
    \item (3) Near keep-out zone
    \item (4) Far keep-out zone
    \item (5) Near obstruction and far keep-out zone
\end{itemize}

An observability scenario is then defined as a 6-tuple of barrier types $b_{\text{direction}}$, representing the selection of barriers in each of the six directions:
\[
    scenario = (b_{x^+}, b_{y^+}, b_{z^+}, b_{x^-}, b_{y^-}, b_{z^-})
\]
For example, the scenario $(0,0,0,0,0,0)$ represents a map point in free space, while $(1,0,0,0,0,0)$ represents a point adjacent to a near obstruction in the $x^+$ direction, such as a shared wall.  

This definition allows for $6^6 = 46,656$ unique observability scenarios. However, not all combinations are valid, and many are equivalent. Scenarios that prevent observation of the map point from any direction (e.g., two near barriers in opposing directions) are removed. These invalid cases can be eliminated by discarding any scenario where $scenario[i] \in \{1, 3, 5\} \land scenario[i+3] \in \{1, 3, 5\}$ for $i \in [0,1,2]$. Reflections about the $x$, $y$, or $z$ axes are considered equivalent and removed, as are rotationally equivalent scenarios. This reduction leaves a total of 896 unique observability scenarios, each representing a distinct observability configuration.

This dataset allows the observability model to be tested across a wide range of conditions, with the unique scenarios serving as ground truth for model evaluation. Scenarios can be queried to determine whether a map point is observable from a given viewpoint, and model accuracy can be measured by comparing predictions against ground-truth observability. \subsubsection{Episodic Visual SLAM Dataset}

To evaluate VBEE performance in a real-world SLAM system, a stereo visual dataset was created consisting of a series of trajectories through a household environment. No dynamic changes occur during the recording of a single trajectory, but environmental changes are introduced between trajectories, matching the episodic run modality described in section \ref{sec:background}. Figure \ref{fig:episodic_datasets} shows example images from each trajectory at roughly the same location.

\begin{figure}[H]
    \centering
    % \includegraphics[width=0.8\textwidth]{resources/placeholder.jpeg}
    \caption{Example images from each trajectory of the episodic visual SLAM dataset}
    \label{fig:episodic_datasets}
\end{figure}

This dataset is intended to replicate realistic episodic SLAM operation in a household environment. Ground truth is difficult to obtain in large-scale 3D environments, so evaluation focuses on the SLAM system's ability to maintain a consistent map across multiple trajectories. LibVBEE is integrated into ORB-SLAM3, and performance is evaluated by measuring its ability to identify and remove outdated map points while preserving map consistency across episodes.
